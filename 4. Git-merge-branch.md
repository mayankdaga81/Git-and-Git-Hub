# 🔀 Git Merge – Quick Revision Notes

---

# 🧠 Scenario

Branches:

- `master`
- `nav-bar`

Task:
👉 Merge `nav-bar` into `master`

---

## 1️⃣ Step 1 – Switch to Target Branch

You must be on the branch **where you want the code to come into**.

```bash
git switch master
# or
git checkout master
```

> Always merge INTO the branch you are currently on.

---

## 2️⃣ Step 2 – Merge the Branch

```bash
git merge nav-bar
```

- This merges `nav-bar` into `master`.

---

# ✅ Case 1: No Conflicts

- Merge happens automatically.
- Terminal shows merge success.
- Code is merged in one go.

---

# ❌ Case 2: Merge Conflicts

If both branches modified the same part of a file:

- Git will show a **merge conflict message** in terminal.
- Files will contain conflict markers like:

```
<<<<<<< HEAD
Your current branch code
=======
Incoming branch code
>>>>>>> nav-bar
```

---

## 3️⃣ Resolve Conflicts

When you open VS Code:

- Use the **Merge Editor** (recommended).
- You’ll see options:
  - Accept Current Change
  - Accept Incoming Change
  - Accept Both Changes
  - Compare Changes

Choose based on requirement.

After resolving:

```bash
git add .
git commit
```

Merge will be completed.

---

# ⚠️ Important Notes

- Only the branch you merged INTO gets updated.
  - If you merged into `master`, only `master` is updated.
- The source branch (e.g., `nav-bar` or `footer`) remains unchanged.
- Sometimes you may need to merge back (reverse merge) to keep branches in sync.

---

# 🔁 Example

To merge `footer` into `master`:

```bash
git switch master
git merge footer
```

---

# 🧩 Important Points (Interview Focus)

- Always switch to the target branch before merging.
- Merge conflicts occur when same lines are modified.
- Git does not delete the source branch automatically.
- Use small, frequent commits to reduce conflicts.
- Test your code after merge.

---

💡 Merging = Combining history of two branches into one.
