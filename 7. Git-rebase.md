# 🔄 Git Rebase – Complete Revision Notes

---

# 🔹 What is Git Rebase?

`git rebase` is an **alternative to merge**.

It is used to:
- Integrate changes from one branch into another.
- Maintain a **clean, linear commit history**.
- Clean up commit structure.

👉 Rebase = Replay your branch commits on top of another branch.

---

# 🧠 Core Concept

Instead of combining histories like `merge`,
Rebase **rewrites history**.

It takes your feature branch commits,
removes them temporarily,
moves the branch pointer,
and then reapplies your commits on top.

---

# 📌 Scenario Example

Branches:

```
A---B---C  (master)
     \
      D---E  (bugfix)
```

- `master` has commits: A, B, C
- `bugfix` branched from B
- `bugfix` has commits: D, E
- Meanwhile, master moved ahead to C

---

# 🔁 Before Rebase

History looks like:

```
A---B---C        (master)
     \
      D---E      (bugfix)
```

---

# ▶️ Command

```bash
git switch bugfix
git rebase master
```

⚠️ Always run rebase on the **feature/child branch**, NOT on master.

---

# 🔁 After Rebase

What happens internally:

1. Git temporarily removes commits D and E.
2. Moves bugfix to C.
3. Replays D and E on top of C.

New history:

```
A---B---C---D'---E'   (bugfix)
```

- D' and E' are new commits (rewritten versions).
- Cleaner, linear history.
- No merge commit created.

---

# 🔥 Merge vs Rebase

## Merge

```
A---B---C
     \     \
      D---E---M
```

- Creates extra merge commit (M).
- History becomes non-linear.

## Rebase

```
A---B---C---D'---E'
```

- Clean linear history.
- No merge commit.

---

# 🚨 VERY IMPORTANT RULE

❌ Do NOT run rebase on `master` or `main`.

If you run:

```bash
git switch master
git rebase bugfix
```

You are rewriting master history.

👉 This can break shared repositories.
👉 Never rebase public/shared branches.

---

# 🧩 Conflict During Rebase

If conflict happens:

Git will pause and show error.

Steps:

1. Fix conflict in files.
2. Add resolved files:

```bash
git add .
```

3. Continue rebase:

```bash
git rebase --continue
```

To cancel rebase:

```bash
git rebase --abort
```

---

# ✨ Interactive Rebase (Cleanup Tool)

Used to clean commits.

```bash
git rebase -i HEAD~3
```

Opens editor:

You can:
- pick (keep commit)
- squash (combine commits)
- reword (edit message)
- drop (delete commit)

Example:

Before:

```
Fix typo
Another typo fix
Final typo fix
```

After squash:

```
Fix typo properly
```

👉 Makes commit history professional.

---

# 🧠 Real-World Workflow Example

1️⃣ Create feature branch

```bash
git checkout -b feature-login
```

2️⃣ Make commits

3️⃣ Meanwhile, master gets updated.

4️⃣ Update your feature branch:

```bash
git switch feature-login
git rebase master
```

5️⃣ Push safely:

If already pushed before:

```bash
git push --force-with-lease
```

⚠️ Never use plain `--force` blindly.

---

# 🎯 When to Use Rebase

✅ Before merging feature branch into master  
✅ To keep clean history  
✅ To update feature branch with latest master  
✅ To squash messy commits  

---

# ❌ When NOT to Use Rebase

❌ On shared public branches  
❌ After pushing to team without coordination  
❌ If you don’t understand history rewriting  

---

# 🔥 Final Understanding

Rebase =  
"Take my work, move it on top of latest base, and pretend I started from there."

Merge =  
"Combine histories as they are."

---

# 📌 Interview Key Points

- Rebase rewrites history.
- Creates linear history.
- No merge commit.
- Always run rebase on feature branch.
- Use interactive rebase to clean commits.
- Be careful with force push.

---

💡 Rebase = History cleaner + Linear workflow tool.
