# 🌍 Open Source Contribution & Pull Requests – Quick Revision Notes

---

# 🧠 Before You Start

✅ Talk to maintainers before starting work.  
✅ Comment on the issue first.  
✅ Avoid duplicate work.  

Open source = Collaboration, not competition.

---

# 📝 Basic Open Source Flow

1️⃣ Find a project  
2️⃣ Open / pick an Issue  
3️⃣ Get the issue assigned  
4️⃣ Work on it  
5️⃣ Create Pull Request  
6️⃣ Iterate based on review  

⚠️ Making a PR is NOT a job guarantee.

---

# 🍴 Step 1 – Create a Fork

- Fork = Copy of someone else's repository into your GitHub account.
- You get your own version to work on.

👉 Fork brings the repo under your GitHub account.

---

# 📥 Step 2 – Clone the Fork

```bash
git clone <URL>
```

- Downloads repo to your local machine.
- Automatically sets `origin` remote.
- No need to manually add origin.

---

# 🌿 Step 3 – Create Your Own Branch

❌ Do NOT work on `main` directly.

Create a feature branch:

```bash
git checkout -b feature-branch-name
```

Best practice:
- `feature-login`
- `fix-navbar-bug`
- `docs-update`

---

# 💻 Step 4 – Work and Add Value

- Make meaningful changes.
- Follow project guidelines.
- Follow coding standards.
- Test your changes.

---

# 🚀 Step 5 – Push Your Branch

```bash
git push -u origin feature-branch-name
```

- Push your branch to your fork.
- Sets upstream tracking.

---

# 🔀 Step 6 – Create Pull Request (PR)

After pushing:

- Go to GitHub.
- You will see "Compare & Pull Request" button.
- Click it.

---

# 🧾 What is a Pull Request?

Pull Request =  
Request to merge your changes into the original repository.

You are saying:

"I have added something valuable. Please review and merge."

---

# 🛠 While Creating PR

You can choose:

- Base Repository (Owner's repo)
- Base Branch (Usually main/master)
- Your Fork
- Your Feature Branch

Example:

```
Base repo: owner/project
Base branch: main
Compare: your-username/project
Compare branch: feature-login
```

---

# ✍️ Write Good PR Description

Important:

- Clear Title
- Detailed Description
- What problem it solves
- Screenshots (if UI change)
- Reference Issue Number (#123)

Good PRs get merged faster.

---

# 🔁 PR Review Process

Maintainer may:

- Approve
- Request changes
- Suggest improvements
- Reject

If changes requested:

```bash
# Make changes
git add .
git commit -m "Address review comments"
git push
```

PR updates automatically.

---

# 🧩 Important Points (Interview Focus)

- Always discuss before working.
- Work on a separate branch.
- Fork → Clone → Branch → Commit → Push → PR
- PR is a request, not an order.
- Maintain professional communication.
- Follow contribution guidelines (README / CONTRIBUTING.md).
- Keep commits clean.

---

# 🔥 Complete Flow Summary

```
Fork → Clone → Create Branch → Make Changes → Commit → Push → Create PR → Review → Merge
```

---

💡 Open Source = Communication + Quality + Collaboration.
